## 1. Core Java

### OOPs

*   **What is the difference between abstraction and encapsulation?**
    *   **Abstraction:** Focuses on *what* an object does rather than *how* it does it. It's about showing essential features and hiding complex implementation details. Achieved using abstract classes and interfaces. (e.g., A car's steering wheel abstracts away the complex steering mechanism).
    *   **Encapsulation:** Focuses on *how* an object is implemented. It's the bundling of data (attributes) and methods that operate on that data into a single unit (class), and restricting direct access to some of the object's components. Achieved using access modifiers (`private`, `protected`, `public`) and providing public getter/setter methods. (e.g., A car's engine is encapsulated; you interact with it via the accelerator, not by directly manipulating pistons).

*   **How is polymorphism achieved in Java?**
    Polymorphism (meaning "many forms") in Java is primarily achieved through:
    *   **Method Overloading (Compile-time/Static Polymorphism):** Multiple methods in the same class have the same name but different parameters (number, type, or order). The compiler decides which method to call based on the arguments provided.
    *   **Method Overriding (Runtime/Dynamic Polymorphism):** A subclass provides a specific implementation for a method that is already defined in its superclass. The JVM decides which method to call at runtime based on the actual object type. This is the more powerful form of polymorphism as it allows treating objects of different classes in a uniform way through a common interface or superclass.

*   **Can you override a static method?**
    No, a static method cannot be overridden in Java. If a subclass declares a static method with the same signature as a static method in its superclass, it's called **method hiding**, not overriding. The method called depends on the type of the reference variable, not the actual object type, which is different from dynamic polymorphism.

*   **Explain method overloading vs overriding with examples.**
    *   **Method Overloading:**
        *   **Definition:** Multiple methods in the *same class* share the same name but have different parameter lists (different number of arguments, different types of arguments, or different order of arguments).
        *   **Polymorphism Type:** Compile-time (Static) Polymorphism.
        *   **Example:**
            ```java
            class Calculator {
                int add(int a, int b) {
                    return a + b;
                }
                double add(double a, double b) { // Overloaded method
                    return a + b;
                }
                int add(int a, int b, int c) { // Overloaded method
                    return a + b + c;
                }
            }
            ```
    *   **Method Overriding:**
        *   **Definition:** A subclass provides its own specific implementation for a method that is already defined in its superclass. The method signature (name, parameters, return type) must be the same or covariant.
        *   **Polymorphism Type:** Runtime (Dynamic) Polymorphism.
        *   **Example:**
            ```java
            class Animal {
                void makeSound() {
                    System.out.println("Animal makes a sound");
                }
            }

            class Dog extends Animal {
                @Override // Good practice to use @Override annotation
                void makeSound() { // Overriding the makeSound method
                    System.out.println("Dog barks");
                }
            }

            // Usage:
            // Animal myAnimal = new Dog();
            // myAnimal.makeSound(); // Output: Dog barks (runtime polymorphism)
            ```

### Collections

*   **Difference between `ArrayList` and `LinkedList`.**

    | Feature           | `ArrayList`                                       | `LinkedList`                                      |
    | :---------------- | :------------------------------------------------ | :------------------------------------------------ |
    | **Data Structure** | Dynamic array                                     | Doubly linked list                                |
    | **Random Access** | Fast (O(1)) using `get(index)`                  | Slow (O(n)) as it requires traversal              |
    | **Insert/Delete** | Slow (O(n)) in the middle/beginning (requires shifting elements) | Fast (O(1)) once position is found                |
    | **Memory**        | Less overhead (stores elements contiguously)      | More overhead (stores elements + next/prev pointers) |
    | **Usage**         | Frequent read operations, less insert/delete      | Frequent insert/delete operations, less read      |

*   **How does `HashMap` work internally?**
    `HashMap` stores data in key-value pairs. Internally, it uses an array of `Node` objects (or `Entry` objects in older Java versions). Each `Node` represents a key-value pair and contains: `hash`, `key`, `value`, and `next` (a pointer to the next node in case of a collision).
    1.  **`put(key, value)`:**
        *   The `hashCode()` of the key is calculated.
        *   This hash code is used to determine the `index` (bucket) in the internal array where the `Node` should be stored. `index = hash % array.length`.
        *   If multiple keys hash to the same index (a **collision**), the `Node`s are stored in a linked list at that array index.
        *   From Java 8, if a linked list at a bucket becomes too long (default threshold of 8 nodes), it converts into a **balanced tree** (Red-Black tree) for better performance (O(log n) instead of O(n) in worst-case linked list traversal).
    2.  **`get(key)`:**
        *   The `hashCode()` of the key is calculated to find the array `index`.
        *   At that index, the `HashMap` iterates through the linked list (or tree) of `Node`s.
        *   For each `Node`, it first compares hash codes, then uses `equals()` method to find the exact key match.
        *   Once the key is found, its corresponding value is returned.
    *   **Load Factor:** When the number of entries exceeds `capacity * loadFactor` (default `0.75`), the `HashMap` `rehashes` (resizes the internal array, usually doubling it) and re-distributes all entries to new buckets. This ensures average O(1) performance.

*   **What is the difference between `HashMap`, `TreeMap`, and `LinkedHashMap`?**

    | Feature           | `HashMap`                                       | `TreeMap`                                       | `LinkedHashMap`                                    |
    | :---------------- | :---------------------------------------------- | :---------------------------------------------- | :------------------------------------------------- |
    | **Order**         | No guaranteed order                             | Sorted by keys (natural order or `Comparator`)  | Insertion order (or access order)                  |
    | **Null Keys/Values** | Allows one null key, multiple null values       | Does not allow null keys (throws `NullPointerException`), allows null values | Allows one null key, multiple null values          |
    | **Implementation** | Hash table (array of linked lists/red-black trees) | Red-Black Tree                                  | Hash table + Doubly linked list                    |
    | **Performance**   | O(1) average for `get`/`put`/`remove`           | O(log n) for `get`/`put`/`remove`               | O(1) average for `get`/`put`/`remove` (slightly slower than `HashMap` due to linked list overhead) |
    | **Use Case**      | Fast lookups, no order requirement              | Sorted data, range queries                      | Maintain insertion order, build LRU caches         |

*   **Fail-fast vs fail-safe iterators.**
    *   **Fail-Fast Iterators:**
        *   Iterators that immediately throw a `ConcurrentModificationException` if the underlying collection is structurally modified (e.g., elements added, removed, or resized) by any means other than the iterator's own `remove()` or `add()` methods, after the iterator has been created.
        *   They are implemented by most non-concurrent collections in `java.util` (e.g., `ArrayList`, `HashMap`).
        *   They are designed to detect concurrent modification issues quickly to prevent unpredictable behavior.
    *   **Fail-Safe Iterators:**
        *   Iterators that do *not* throw an exception if the underlying collection is modified during iteration. They operate on a copy or snapshot of the collection at the time the iterator was created.
        *   They are implemented by concurrent collections in `java.util.concurrent` (e.g., `ConcurrentHashMap`, `CopyOnWriteArrayList`).
        *   While they prevent `ConcurrentModificationException`, they might not show the most up-to-date state of the collection.

### Exception Handling

*   **Difference between checked and unchecked exceptions.**

    | Feature          | Checked Exceptions                               | Unchecked Exceptions                               |
    | :--------------- | :----------------------------------------------- | :------------------------------------------------- |
    | **Compile-time** | Must be handled (caught or declared with `throws`) | Not required to be handled                         |
    | **Inheritance**  | Subclasses of `Exception` (but not `RuntimeException`) | Subclasses of `RuntimeException` or `Error`        |
    | **Recovery**     | Recoverable errors (e.g., file not found)        | Programming bugs or logical errors (e.g., `NullPointerException`) |
    | **Example**      | `IOException`, `SQLException`, `ClassNotFoundException` | `NullPointerException`, `ArrayIndexOutOfBoundsException`, `ArithmeticException` |

*   **What is `finally` block used for?**
    The `finally` block is used to execute code that *must* run, regardless of whether an exception occurred in the `try` block or was caught by a `catch` block. It is typically used for cleanup operations, such as:
    *   Closing resources (file streams, database connections, network sockets).
    *   Releasing locks.
    *   Ensuring that a specific piece of code always executes.

*   **Can you catch multiple exceptions in a single catch block?**
    Yes, from Java 7 onwards, you can catch multiple exceptions in a single `catch` block using the multi-catch feature. The exception types are separated by a vertical bar (`|`).
    ```java
    try {
        // code that might throw exceptions
    } catch (IOException | SQLException e) {
        System.err.println("An I/O or SQL error occurred: " + e.getMessage());
    }
    ```

*   **Custom exception creation with an example.**
    You can create custom exceptions by extending `Exception` (for checked exceptions) or `RuntimeException` (for unchecked exceptions).
    ```java
    // Custom Checked Exception
    class InvalidAgeException extends Exception {
        public InvalidAgeException(String message) {
            super(message);
        }
    }

    // Custom Unchecked Exception
    class InsufficientFundsException extends RuntimeException {
        public InsufficientFundsException(String message) {
            super(message);
        }
    }

    public class ExceptionDemo {
        public static void validateAge(int age) throws InvalidAgeException {
            if (age < 18) {
                throw new InvalidAgeException("Age must be 18 or older.");
            }
            System.out.println("Age is valid.");
        }

        public static void withdraw(double amount, double balance) {
            if (amount > balance) {
                throw new InsufficientFundsException("Insufficient funds. Available: " + balance);
            }
            System.out.println("Withdrawal successful. Remaining balance: " + (balance - amount));
        }

        public static void main(String[] args) {
            try {
                validateAge(16); // This will throw InvalidAgeException
            } catch (InvalidAgeException e) {
                System.err.println("Error: " + e.getMessage());
            }

            try {
                withdraw(200, 150); // This will throw InsufficientFundsException
            } catch (InsufficientFundsException e) {
                System.err.println("Error: " + e.getMessage());
            }
        }
    }
    ```

### Multithreading

*   **Difference between `Runnable` and `Thread`.**

    | Feature        | `Runnable`                                  | `Thread`                                        |
    | :------------- | :------------------------------------------ | :---------------------------------------------- |
    | **Type**       | Interface (`java.lang.Runnable`)            | Class (`java.lang.Thread`)                      |
    | **Inheritance** | Allows class to extend another class        | Consumes the single inheritance slot            |
    | **Resource**   | Represents a task to be executed            | Represents a thread of execution                |
    | **Creation**   | `new Thread(new MyRunnable()).start()`      | `new MyThread().start()`                        |
    | **Flexibility** | More flexible, promotes composition         | Less flexible due to inheritance                |
    | **Best Practice** | Generally preferred, as it separates the task from the thread. | Use when you need to extend `Thread` for specific Thread-related functionalities. |

*   **What is a race condition and how do you prevent it?**
    *   **Race Condition:** Occurs when multiple threads try to access and modify shared data concurrently, and the final outcome depends on the unpredictable order of execution of these threads. This often leads to incorrect or inconsistent results.
    *   **Prevention:**
        *   **Synchronization:** Use the `synchronized` keyword (methods or blocks) to ensure that only one thread can access a critical section of code at a time.
        *   **Locks:** Use explicit `Lock` objects (e.g., `ReentrantLock` from `java.util.concurrent.locks`) for more fine-grained control over locking.
        *   **Atomic Operations:** Use `java.util.concurrent.atomic` classes (e.g., `AtomicInteger`, `AtomicLong`) for atomic operations on single variables without explicit locking.
        *   **Concurrent Collections:** Use thread-safe collections from `java.util.concurrent` (e.g., `ConcurrentHashMap`, `CopyOnWriteArrayList`).
        *   **Immutability:** Make shared data immutable. Immutable objects are inherently thread-safe.

*   **Synchronized vs volatile keyword.**

    | Feature          | `synchronized`                                    | `volatile`                                     |
    | :--------------- | :------------------------------------------------ | :--------------------------------------------- |
    | **Purpose**      | Ensures atomicity and mutual exclusion            | Ensures visibility of variable changes         |
    | **Scope**        | Methods or code blocks                            | Variables (fields)                             |
    | **Locking**      | Acquires and releases an intrinsic lock           | No explicit locking involved                   |
    | **Memory Barrier** | Yes (both read and write barriers)                | Yes (read and write barriers)                  |
    | **Thread-Safety** | Provides atomicity for a block of code, prevents race conditions | Ensures changes are visible to other threads, but does not guarantee atomicity for compound operations |
    | **Cost**         | Higher overhead (locking/unlocking)               | Lower overhead                                 |
    | **Use Case**     | Protecting critical sections, compound operations | Ensuring visibility of a single variable, flag |

*   **Thread lifecycle and `ExecutorService` basics.**
    *   **Thread Lifecycle:** A thread goes through several states during its lifetime:
        *   **New:** A thread has been created but not yet started (`new Thread()`).
        *   **Runnable:** A thread that is ready to run and waiting for CPU time (after `start()` is called). It might be running or waiting in the run queue.
        *   **Blocked:** A thread that is temporarily inactive because it's waiting for a monitor lock (e.g., entering a `synchronized` block that's already locked).
        *   **Waiting:** A thread that is indefinitely waiting for another thread to perform a particular action (e.g., calling `wait()`, `join()`, `LockSupport.park()`).
        *   **Timed Waiting:** A thread that is waiting for a specified amount of time or until another thread performs an action (e.g., calling `sleep(long)`, `wait(long)`, `join(long)`).
        *   **Terminated:** A thread that has completed its execution or has been abnormally terminated.
    *   **`ExecutorService` Basics:**
        *   `ExecutorService` is an interface in `java.util.concurrent` that provides a higher-level abstraction for managing threads and tasks. Instead of creating and managing `Thread` objects directly, you submit `Runnable` or `Callable` tasks to an `ExecutorService`.
        *   **Benefits:**
            *   **Thread Pool Management:** It manages a pool of worker threads, reducing the overhead of creating and destroying threads for each task.
            *   **Task Submission:** Provides methods like `execute(Runnable)` and `submit(Callable/Runnable)` to submit tasks. `submit` returns a `Future` object for managing the task's result or status.
            *   **Lifecycle Management:** Allows graceful shutdown of the thread pool.
        *   **Common Implementations:**
            *   `Executors.newFixedThreadPool(n)`: Creates a thread pool with a fixed number of threads.
            *   `Executors.newCachedThreadPool()`: Creates a thread pool that creates new threads as needed, but reuses existing threads if available.
            *   `Executors.newSingleThreadExecutor()`: Creates a single-threaded executor.

### Streams

*   **Filter all employees with salary > 50K using streams.**
    ```java
    import java.util.Arrays;
    import java.util.List;
    import java.util.stream.Collectors;

    class Employee {
        String name;
        double salary;
        public Employee(String name, double salary) {
            this.name = name;
            this.salary = salary;
        }
        public double getSalary() { return salary; }
        @Override
        public String toString() { return name + "(" + salary + ")"; }
    }

    public class StreamFilterExample {
        public static void main(String[] args) {
            List<Employee> employees = Arrays.asList(
                new Employee("Alice", 60000),
                new Employee("Bob", 45000),
                new Employee("Charlie", 75000),
                new Employee("David", 50000)
            );

            List<Employee> highPaidEmployees = employees.stream()
                .filter(emp -> emp.getSalary() > 50000)
                .collect(Collectors.toList());

            System.out.println(highPaidEmployees);
        }
    }
    ```

*   **Group a list of users by city using `Collectors.groupingBy`.**
    ```java
    import java.util.Arrays;
    import java.util.List;
    import java.util.Map;
    import java.util.stream.Collectors;

    class User {
        String name;
        String city;
        public User(String name, String city) {
            this.name = name;
            this.city = city;
        }
        public String getCity() { return city; }
        @Override
        public String toString() { return name + " (" + city + ")"; }
    }

    public class StreamGroupingExample {
        public static void main(String[] args) {
            List<User> users = Arrays.asList(
                new User("Alice", "New York"),
                new User("Bob", "London"),
                new User("Charlie", "New York"),
                new User("David", "Paris"),
                new User("Eve", "London")
            );

            Map<String, List<User>> usersByCity = users.stream()
                .collect(Collectors.groupingBy(User::getCity));

            usersByCity.forEach((city, userList) -> {
                System.out.println(city + ": " + userList);
            });
        }
    }
    ```

*   **Difference between `map()` and `flatMap()`?**
    *   **`map()`:** Transforms each element of a stream into a *single* new element. It's a one-to-one transformation. The output stream will have the same number of elements as the input stream.
        *   Example: `Stream<String>` to `Stream<Integer>` (length of string).
    *   **`flatMap()`:** Transforms each element of a stream into *zero or more* elements, and then flattens the resulting streams into a single stream. It's a one-to-many or one-to-zero transformation, followed by flattening.
        *   Example: `Stream<List<String>>` to `Stream<String>` (flattening a list of lists). Or `Stream<Sentence>` to `Stream<Word>`.

    ```java
    import java.util.Arrays;
    import java.util.List;
    import java.util.stream.Collectors;
    import java.util.stream.Stream;

    public class MapFlatMapExample {
        public static void main(String[] args) {
            List<List<String>> listOfLists = Arrays.asList(
                Arrays.asList("a", "b"),
                Arrays.asList("c", "d", "e"),
                Arrays.asList("f")
            );

            // Using map: Result is Stream<Stream<String>> (Stream of lists)
            List<Stream<String>> mappedStream = listOfLists.stream()
                .map(list -> list.stream()) // Each list becomes a stream
                .collect(Collectors.toList());
            System.out.println("Mapped Stream (Stream of Streams): " + mappedStream);

            // Using flatMap: Result is Stream<String> (single flattened stream)
            List<String> flatMappedList = listOfLists.stream()
                .flatMap(list -> list.stream()) // Each list's stream is flattened into the main stream
                .collect(Collectors.toList());
            System.out.println("FlatMapped List: " + flatMappedList);
        }
    }
    ```

### Lambdas

*   **Syntax of a lambda expression.**
    A lambda expression is a short block of code that takes parameters and returns a value. It's often used to implement functional interfaces.
    *   **Basic Syntax:** `(parameters) -> expression` or `(parameters) -> { statements; }`
    *   **Examples:**
        *   `() -> System.out.println("Hello")` (No parameters, single statement)
        *   `(a, b) -> a + b` (Two parameters, single expression/return value)
        *   `name -> System.out.println("Hello, " + name)` (Single parameter, type inference)
        *   `(int x, int y) -> { int sum = x + y; return sum; }` (Explicit types, multiple statements)

*   **When would you prefer using lambda over anonymous class?**
    You would prefer using lambda expressions over anonymous classes when:
    *   **Conciseness:** Lambdas are much more concise and readable for implementing single-method interfaces.
    *   **Readability:** They reduce boilerplate code, making the intent clearer.
    *   **Functional Style:** They enable a more functional programming style, especially with the Stream API.
    *   **No `this` Shadowing:** Inside a lambda, `this` refers to the enclosing instance, unlike anonymous classes where `this` refers to the anonymous class instance itself.
    *   **Performance:** While often similar, in some cases, lambdas can lead to better performance due to JVM optimizations.

    However, anonymous classes might still be preferred for:
    *   Implementing an interface with multiple abstract methods.
    *   Implementing an abstract class.
    *   Adding fields or additional methods to the implementation.

*   **Convert a functional interface method to a lambda.**
    A functional interface is an interface with exactly one abstract method.
    *   **Original functional interface usage (pre-Java 8 or anonymous class):**
        ```java
        interface MyConverter {
            int convert(String s);
        }

        // ...
        MyConverter converter = new MyConverter() {
            @Override
            public int convert(String s) {
                return Integer.parseInt(s);
            }
        };
        int number = converter.convert("123");
        ```
    *   **Using a lambda expression:**
        ```java
        interface MyConverter {
            int convert(String s);
        }

        // ...
        MyConverter converter = (s) -> Integer.parseInt(s);
        int number = converter.convert("123");

        // Or using method reference for even more conciseness
        MyConverter converterRef = Integer::parseInt;
        int numberRef = converterRef.convert("456");
        ```

### Functional Interfaces

*   **What is a functional interface?**
    A functional interface is an interface that contains **exactly one abstract method**. It can have any number of default and static methods. The `@FunctionalInterface` annotation is optional but recommended to ensure that the interface adheres to the functional interface contract. They are key to enabling lambda expressions and method references.

*   **Give examples of built-in functional interfaces in Java 8: `Predicate`, `Function`, `Consumer`, `Supplier`.**
    *   **`Predicate<T>`:** Represents a predicate (boolean-valued function) of one argument.
        *   **Abstract method:** `boolean test(T t);`
        *   **Use case:** Filtering elements in a stream.
        *   Example: `Predicate<Integer> isEven = n -> n % 2 == 0;`
    *   **`Function<T, R>`:** Represents a function that accepts one argument and produces a result.
        *   **Abstract method:** `R apply(T t);`
        *   **Use case:** Mapping elements from one type to another.
        *   Example: `Function<String, Integer> stringToInt = s -> Integer.parseInt(s);`
    *   **`Consumer<T>`:** Represents an operation that accepts a single input argument and returns no result.
        *   **Abstract method:** `void accept(T t);`
        *   **Use case:** Performing an action on each element (e.g., in `forEach`).
        *   Example: `Consumer<String> printUpperCase = s -> System.out.println(s.toUpperCase());`
    *   **`Supplier<T>`:** Represents a supplier of results. It takes no arguments and returns a result.
        *   **Abstract method:** `T get();`
        *   **Use case:** Lazy initialization, factory methods, providing default values.
        *   Example: `Supplier<Double> randomNumber = () -> Math.random();`

*   **What does `@SpringBootApplication` include?**
    `@SpringBootApplication` is a convenience annotation that combines three commonly used Spring Boot annotations:
    *   `@Configuration`: Tags the class as a source of bean definitions for the application context.
    *   `@EnableAutoConfiguration`: Tells Spring Boot to start adding beans based on classpath settings, other beans, and various property settings. This is the magic that makes Spring Boot "just work."
    *   `@ComponentScan`: Tells Spring to look for other components, configurations, and services in the specified package (by default, the package where `@SpringBootApplication` is located and its sub-packages), allowing it to find controllers, services, and repositories.
